# Smart Ticket Router

An AI-powered support ticket classifier: paste a raw support ticket into a dashboard and get back a structured routing decision — category, priority, assigned team, customer tone, and a one-sentence justification — generated by OpenAI's `gpt-4.1-mini` using Structured Outputs.

## Why this exists

Manually triaging support tickets means a human reads each one, decides what it's about, how urgent it is, and which team should own it — repeated for every ticket that comes in. This project automates that first triage step: the AI reads the ticket and returns the same structured decision a human would, in a fraction of the time (see [Manual vs AI Timing](#manual-vs-ai-timing) below), freeing support staff to focus on actually resolving issues instead of sorting them.

## Architecture

```
User (browser dashboard)
    -> Flask app.py            (routes, input validation, JSON API)
    -> router.py                (prompt building, response parsing/validation, retry, fallback)
    -> ai_client.py             (OpenAI API wrapper, Structured Outputs enforcement)
    -> OpenAI gpt-4.1-mini
    -> JSON response (schema-enforced by the API itself)
    -> router.py                (defense-in-depth: re-validated against required keys + allowed enum values)
    -> app.py -> JSON (dashboard) or templates/index.html (no-JS fallback)
```

| File | Responsibility |
|---|---|
| `app.py` | Flask routes: `/` (classic form + no-JS fallback), `/api/route` (JSON API used by the dashboard), `/api/benchmark` (runs the timing comparison on demand), `/api/sample-tickets` (serves the demo dataset to the UI). Rejects empty or over-length (>5000 char) tickets before calling the router. |
| `router.py` | Builds the prompt, calls the AI client, parses the JSON response, validates it (required keys present + values are legal enum members), retries once on an invalid response, and falls back to a safe default result if still invalid |
| `ai_client.py` | Thin wrapper around the OpenAI client; requests Structured Outputs (`response_format`) so the API enforces the schema at generation time; converts any API-level failure into a typed `AIServiceError` |
| `prompts.py` | The system prompt (schema, categories, priorities, teams, tone, priority-assignment rules, language handling) plus the `CATEGORIES` / `PRIORITIES` / `TEAMS` / `TONES` constants and the `TICKET_JSON_SCHEMA` used for both output validation and Structured Outputs |
| `templates/index.html`, `static/style.css`, `static/app.js` | The dashboard UI — ticket input, routing result, live analytics, ticket buckets, searchable history, sample ticket library, and the manual-vs-AI benchmark panel. `static/app.js` drives all of it via `fetch()` calls to the JSON API; `templates/index.html` also renders a working result server-side for the no-JS fallback path. |
| `sample_tickets.json` | 35 example tickets spanning all 5 categories plus tagged edge cases (`angry`, `very_short`, `ambiguous`, `non_english`, `long`); browsable/insertable directly from the dashboard's Sample Ticket Library |
| `timing_benchmark.py` | Measures real AI response time across all 35 sample tickets and compares it against an estimated manual-triage baseline; runnable from the CLI or from the dashboard's "Run Benchmark" button |

## Output schema

Every routing result is this JSON shape, enforced by OpenAI's Structured Outputs mode (not just prompt wording — see [Prompt engineering](#prompt-engineering)):

```json
{
  "category": "Billing | Technical | Account | Feature Request | General Inquiry",
  "priority": "High | Medium | Low",
  "assigned_team": "Billing Team | Technical Support | Account Team | Product Team | Customer Support",
  "reasoning": "one short sentence",
  "tone": "Angry | Frustrated | Neutral | Confused"
}
```

**Note on `tone`:** this is a 5th field beyond the 4 originally specified (`category`/`priority`/`assigned_team`/`reasoning`). It was added later as an enhancement to surface the customer's emotional state in the dashboard. All 4 original required fields are unchanged and still always present — `tone` is purely additive — but it does mean the live schema is a superset of the originally specified one.

## API endpoints

| Endpoint | Method | Purpose |
|---|---|---|
| `/` | GET / POST | Classic form UI; POST works even with JavaScript disabled (renders the result server-side) |
| `/api/route` | POST | Body `{"ticket": "..."}` → routing result JSON. Used by the dashboard for async submission (loading spinner, no page reload). |
| `/api/benchmark` | POST | Runs `timing_benchmark.run_benchmark()` against all 35 sample tickets and returns the summary stats. Triggered manually from the dashboard since it makes 35 real API calls (~1 minute). |
| `/api/sample-tickets` | GET | Returns the contents of `sample_tickets.json`, used to populate the dashboard's Sample Ticket Library. |

## Prompt engineering

Reliability here comes from two layers working together: prompt design, and API-level schema enforcement.

- **Explicit schema + closed enums** for `category`, `priority`, `assigned_team`, and `tone`, so the model is choosing from a fixed list rather than free-typing labels — described in the prompt text *and* encoded as an actual JSON Schema (`prompts.TICKET_JSON_SCHEMA`).
- **Structured Outputs (`response_format`)**: `ai_client.py` passes `TICKET_JSON_SCHEMA` to OpenAI's `response_format` parameter with `strict: true`. This makes the API itself guarantee schema-conformant JSON (correct keys, correct enum values) at generation time, rather than relying on the model choosing to follow instructions. Verified live: `gpt-4.1-mini` supports this, and post-rollout testing showed every response valid on the first attempt.
- **Priority Guidance block**: added after testing showed the model would sometimes assign `High` priority to vague, low-information tickets (e.g. a 2-word ticket like *"Refund please."* was originally marked `High` with reasoning that only asserted urgency, without evidence). The prompt now defines concretely what qualifies as High/Medium/Low and requires the reasoning sentence to cite specific evidence from the ticket text — not just assert a priority.
- **Tone field**: classifies the customer's emotional state (`Angry`/`Frustrated`/`Neutral`/`Confused`) separately from `priority`, so urgency and emotion aren't conflated — a calm-but-urgent outage report and an angry-but-low-impact complaint get different tones but are prioritized on impact, not tone alone.
- **Explicit language handling**: the ticket may be written in any language; the model is instructed to classify it the same way regardless, and to always respond using the exact English enum labels. (Verified against Spanish and French sample tickets.)

## AI flow

1. User submits ticket text via the dashboard (async `fetch` to `/api/route`) or the classic form.
2. `router.build_prompt()` wraps the ticket in the system prompt.
3. `ai_client.get_ai_response()` calls OpenAI with `temperature=0` and Structured Outputs enforced, raising `AIServiceError` on any API-level failure (this is **not** retried — a down API won't be hammered with repeat calls).
4. `router._parse_and_validate()` strips markdown code fences (a defensive habit from before Structured Outputs was added — models occasionally still wrap JSON in ` ```json ` blocks), parses the JSON, and validates both structure (all 5 required keys present) and content (values are legal enum members).
5. If parsing/validation fails, `router.route_ticket()` retries once (`MAX_ATTEMPTS = 2` total) with a fresh API call before giving up.
6. If everything fails — API error, or both attempts producing invalid output — the router returns a safe fallback result (`category: Unknown`, `priority: Low`, `assigned_team: Customer Support`, `tone: Neutral`) rather than surfacing a broken response or crashing.

## Manual vs AI timing

`timing_benchmark.py` runs all 35 sample tickets through the real AI pipeline and times each call, then compares it to an estimated manual-triage time (varied by ticket type — a 2-word ticket is faster to read than a long, detailed bug report). Run it from the CLI, or click **Run Benchmark** in the dashboard's "Manual vs AI Timing" panel:

```bash
python timing_benchmark.py
```

Latest run:

| | AI (measured) | Manual (estimated) |
|---|---|---|
| Total, 35 tickets | 46.33s | 5,600s (~93 min) |
| Average per ticket | 1.32s | 160s |
| **Speedup** | **~121x** | |

**Important caveat:** the manual-side numbers are a documented estimate based on typical helpdesk triage times (see the constants in `timing_benchmark.py`), not a measurement of an actual human agent working through these tickets. The AI-side numbers are real, measured latency.

## Dashboard features

The UI (`templates/index.html` + `static/app.js` + `static/style.css`) is a single-page dashboard, not just a form:

- Light/dark theme (persisted), responsive layout, loading spinner + disabled button while a ticket is in flight
- Routing result shown as color-coded badges (category, priority — red/orange/green, team, tone), plus a collapsible raw JSON viewer with copy/download buttons
- Live analytics (totals by category and priority), ticket buckets grouped by category, and a searchable/filterable session history — all derived from one in-memory ticket log, reset together
- **Sample Ticket Library**: browse and insert any of the 35 tickets from `sample_tickets.json` directly, filterable by edge-case tag, via `/api/sample-tickets`
- **Manual vs AI Timing panel**: a "Run Benchmark" button triggers `/api/benchmark` and renders the comparison (stat tiles + proportional bars) live in the dashboard — not just in a terminal
- Toast notifications distinguish three failure modes: a validation error (empty/too-long ticket), an "AI service unavailable" fallback, and an "AI returned invalid JSON" fallback — using the different `reasoning` text each path already produces, no extra backend signaling needed
- Everything above degrades gracefully: the `/` route still works as a classic form POST with server-rendered results if JavaScript is disabled

## Error handling

| Scenario | Behavior |
|---|---|
| Empty ticket | Rejected with a friendly inline error; form re-rendered with the (empty) input |
| Ticket over 5,000 characters | Rejected with a friendly inline error; original text preserved in the textarea |
| OpenAI API failure (timeout, auth, rate limit, etc.) | Caught as `AIServiceError`; **not retried**; user sees a fallback result flagged for manual routing rather than a stack trace |
| Malformed / non-JSON AI response, or one with missing/illegal field values | Retried once with a fresh API call; if still invalid, falls back to the same safe default result |
| Ticket in a non-English language | Classified normally; output is always the standard English labels |

## Known limitations

- **Manual timing is an estimate**, not measured from a real support agent (see caveat above).
- **The "angry ⇒ High priority" heuristic isn't absolute.** A ticket that's angry in tone but describes no concrete unresolved problem (e.g. "I've emailed three times and no one has responded") can be scored `Medium` under the current guidance, since anger alone without a described impact doesn't meet the High-priority bar. This is a deliberate trade-off, not a bug.
- **Validation checks structure, not truth.** The router confirms the reasoning field is present and the enum values are legal; it can't verify that the reasoning sentence is actually a correct explanation.
- **Retry is bounded and same-prompt.** `MAX_ATTEMPTS = 2` means one retry, with the identical prompt — it doesn't reformulate the request or add "your last response was invalid" context, so it wouldn't help against a persistently malformed model.
- **The `tone` field extends the schema beyond the originally specified 4 fields** (see [Output schema](#output-schema)) — a deliberate, documented addition, not an oversight.
