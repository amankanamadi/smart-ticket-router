# Smart Ticket Router

An AI-powered support ticket classifier: paste a raw support ticket into a web form and get back a structured routing decision — category, priority, assigned team, and a one-sentence justification — generated by OpenAI's `gpt-4.1-mini`.

## Why this exists

Manually triaging support tickets means a human reads each one, decides what it's about, how urgent it is, and which team should own it — repeated for every ticket that comes in. This project automates that first triage step: the AI reads the ticket and returns the same structured decision a human would, in a fraction of the time (see [Manual vs AI Timing](#manual-vs-ai-timing) below), freeing support staff to focus on actually resolving issues instead of sorting them.

## Architecture

```
User (browser)
    -> Flask app.py            (form handling, input validation)
    -> router.py                (prompt building, response parsing/validation, fallback)
    -> ai_client.py             (OpenAI API wrapper)
    -> OpenAI gpt-4.1-mini
    -> JSON response
    -> router.py                (validated against required keys + allowed enum values)
    -> app.py -> templates/index.html   (rendered result)
```

| File | Responsibility |
|---|---|
| `app.py` | Flask routes; rejects empty or over-length (>5000 char) tickets with a friendly inline error before calling the router |
| `router.py` | Builds the prompt, calls the AI client, parses the JSON response, validates it (required keys present + values are legal enum members), and falls back to a safe default result on any failure |
| `ai_client.py` | Thin wrapper around the OpenAI client; converts any API-level failure into a typed `AIServiceError` |
| `prompts.py` | The system prompt (schema, categories, priorities, teams, priority-assignment rules, language handling) plus the `CATEGORIES` / `PRIORITIES` / `TEAMS` constants used for output validation |
| `templates/index.html`, `static/style.css` | Minimal, non-technical-friendly UI: textarea in, result card out |
| `sample_tickets.json` | 35 example tickets spanning all 5 categories plus tagged edge cases (`angry`, `very_short`, `ambiguous`, `non_english`, `long`) |
| `timing_benchmark.py` | Measures real AI response time across all 35 sample tickets and compares it against an estimated manual-triage baseline |

## Output schema

Every routing result is this fixed JSON shape:

```json
{
  "category": "Billing | Technical | Account | Feature Request | General Inquiry",
  "priority": "High | Medium | Low",
  "assigned_team": "Billing Team | Technical Support | Account Team | Product Team | Customer Support",
  "reasoning": "one short sentence"
}
```

## Prompt engineering

The system prompt (`prompts.py`) does more than ask for JSON — it constrains the model's answer space and forces its reasoning to be checkable:

- **Explicit schema + closed enums** for `category`, `priority`, and `assigned_team`, so the model is choosing from a fixed list rather than free-typing labels.
- **Priority Guidance block**: added after testing showed the model would sometimes assign `High` priority to vague, low-information tickets (e.g. a 2-word ticket like *"Refund please."* was originally marked `High` with reasoning that only asserted urgency, without evidence). The prompt now defines concretely what qualifies as High/Medium/Low and requires the reasoning sentence to cite specific evidence from the ticket text — not just assert a priority.
- **Explicit language handling**: the ticket may be written in any language; the model is instructed to classify it the same way regardless, and to always respond using the exact English enum labels. (Verified against Spanish and French sample tickets.)

## AI flow

1. User submits ticket text via the form.
2. `router.build_prompt()` wraps the ticket in the system prompt.
3. `ai_client.get_ai_response()` calls OpenAI (`temperature=0` for consistency) and raises `AIServiceError` on any API failure.
4. `router.route_ticket()` strips markdown code fences (models sometimes wrap JSON in ` ```json ` blocks despite instructions), parses the JSON, and validates both structure (all 4 required keys present) and content (values are legal enum members).
5. If anything fails at any step — API error, invalid JSON, missing keys, or illegal enum values — the router returns a safe fallback result (`category: Unknown`, `priority: Low`, `assigned_team: Customer Support`) rather than surfacing a broken response or crashing.

## Manual vs AI timing

`timing_benchmark.py` runs all 35 sample tickets through the real AI pipeline and times each call, then compares it to an estimated manual-triage time (varied by ticket type — a 2-word ticket is faster to read than a long, detailed bug report). Run it yourself:

```bash
python timing_benchmark.py
```

Latest run:

| | AI (measured) | Manual (estimated) |
|---|---|---|
| Total, 35 tickets | 51.04s | 5,600s (~93 min) |
| Average per ticket | 1.46s | 160s |
| **Speedup** | **~110x** | |

**Important caveat:** the manual-side numbers are a documented estimate based on typical helpdesk triage times (see the constants in `timing_benchmark.py`), not a measurement of an actual human agent working through these tickets. The AI-side numbers are real, measured latency.

## Error handling

| Scenario | Behavior |
|---|---|
| Empty ticket | Rejected client-side-visible inline error; form re-rendered with the (empty) input |
| Ticket over 5,000 characters | Rejected with a friendly inline error; original text preserved in the textarea |
| OpenAI API failure (timeout, auth, rate limit, etc.) | Caught as `AIServiceError`; user sees a fallback result flagged for manual routing rather than a stack trace |
| Malformed / non-JSON AI response | Falls back to the same safe default result |
| AI response missing required keys, or using an illegal category/priority/team value | Falls back to the same safe default result |
| Ticket in a non-English language | Classified normally; output is always the standard English labels |

## Known limitations

- **No API-level schema enforcement yet.** Validity is currently enforced by prompt instructions plus post-hoc key/enum checks in `router.py`, not by OpenAI's structured-output/JSON-schema mode. In practice this has been reliable in testing, but it isn't a hard guarantee.
- **No retry on a bad response yet.** A single malformed or invalid AI response goes straight to the generic fallback rather than getting a second attempt.
- **Manual timing is an estimate**, not measured from a real support agent (see caveat above).
- **The "angry ⇒ High priority" heuristic isn't absolute.** A ticket that's angry in tone but describes no concrete unresolved problem (e.g. "I've emailed three times and no one has responded") can be scored `Medium` under the current guidance, since anger alone without a described impact doesn't meet the High-priority bar. This is a deliberate trade-off, not a bug.
- **Validation checks structure, not truth.** The router confirms the reasoning field is present and the enum values are legal; it can't verify that the reasoning sentence is actually a correct explanation.

## Roadmap

- Enforce output schema via OpenAI's structured-output (`response_format`) mode, with the current prompt/validation approach as a fallback layer.
- Add a bounded retry (once) on malformed/invalid responses before falling back.
